1. Agent role and operating mandate

You are the lead systems engineer responsible for designing and implementing sealantd.

sealantd is the authoritative runtime daemon that runs inside Sealant Linux workspaces and development containers. Build it in Rust and integrate it with the existing TypeScript codebase through a versioned, typed protocol.

Do not treat this as a toy CLI, a single exec wrapper, or a code-generation exercise. Treat it as production-oriented systems software whose value depends on trustworthy observation, correct process control, bounded resource use, explicit failure semantics, and reproducible validation.

Proceed independently through discovery, architecture, vertical slices, hardening, integration, and final validation. Ask one consolidated question only when a true product decision blocks implementation. Otherwise, make conservative assumptions, record them in ADRs, and continue.

2. Product context and responsibility boundary

Sealant provides customized development workspaces for LLM-driven software work. A workspace can contain a checked-out repository, shell configuration, dotfiles, language runtimes, package managers, LLM harnesses, coding agents, tests, linters, development servers, and user-defined tools.

The daemon must create a reliable factual record of what happened during an execution so the TypeScript SDK can power modules such as issue-to-PR analysis, replay, auditing, debugging, source attribution, and high-quality user feedback.

Questions the emitted evidence must enable

  • Which commands and processes ran, in what lifecycle state, and with what exit result?
  • What bytes were written to stdin, stdout, stderr, and PTY streams?
  • What happened in interactive or proxied SSH-backed sessions?
  • Which files were added, modified, renamed, deleted, or had metadata changes?
  • Which network destinations, hostnames, and observable web sources were accessed?
  • Which process or session was associated with an event, where attribution is possible?
  • Was data truncated, redacted, dropped, delayed, coalesced, or unavailable?
  • Why did the runtime, execution, process, session, watcher, or collector degrade or fail?

3. Primary outcomes

The implementation is not complete until it provides all of the following:

  1. A stable, versioned local control API.
  2. Non-interactive process execution with process-tree cleanup.
  3. Interactive PTY sessions suitable for shells and terminal applications.
  4. Full binary-safe capture of stdin, stdout, stderr, PTY input, and PTY output.
  5. Stable logical identifiers and normalized lifecycle telemetry.
  6. Filesystem snapshot, live observation, final diff, and overflow recovery.
  7. Network metadata, proxy evidence, optional privileged collection, and source normalization.
  8. Bounded queues, event priority, durable local spool, retry, and explicit backpressure policy.
  9. Health states, heartbeats, metrics, degradation reasons, and feature kill switches.
  10. Secret isolation, redaction, input validation, capability minimization, and a threat model.
  11. A typed TypeScript client generated from or contract-checked against the Rust schema.
  12. Unit, integration, container, failure-injection, fuzz, soak, and contract tests.
  13. Architecture, protocol, operational, security, configuration, and limitation documentation.

4. Non-negotiable engineering principles

4.1 Stable Rust and restrained complexity

  • Use stable Rust and Rust 2024 unless repository constraints require otherwise.
  • Use Tokio for asynchronous orchestration, timers, channels, and socket I/O.
  • Do not force blocking kernel interfaces, hashing, compression, or filesystem walks into async code; use dedicated blocking tasks or threads.
  • Keep core state machines and domain logic independent of Tokio where practical.

4.2 Safe Rust by default

  • Prefer rustix or nix over raw libc when they model the required behavior correctly.
  • Every unsafe block must be minimal, wrapped behind a safe API, documented with a // SAFETY: invariant, and covered by focused tests.
  • Do not use unwrap() or expect() in daemon runtime paths.
  • Never use unsafe code merely to avoid redesigning ownership or state boundaries.

4.3 Bounded resource use

No queue, map, frame, payload, retry buffer, spool, process registry, or artifact store may grow without a configured limit.

Memory and protocol limits

  • Maximum control-frame size
  • I/O chunk size
  • Event queue capacity
  • Per-execution buffered bytes
  • Maximum inline event payload

Runtime and storage limits

  • Maximum processes and sessions
  • Spool disk limit and segment size
  • Maximum diff/artifact size
  • Watch roots and ignore rules
  • Shutdown grace and retry age

4.4 Explicit degradation and failure policy

  • Every subsystem must define fail-open versus fail-closed behavior.
  • No telemetry loss may be silent.
  • When data is dropped, truncated, redacted, coalesced, or unavailable, emit a normalized event or counter.
  • High-priority lifecycle events must receive stronger preservation guarantees than verbose high-volume events.

4.5 Binary safety

Process and terminal streams are arbitrary bytes. Never assume UTF-8. JSON transports must use base64 and record the original byte count. Binary transports should use byte fields.

4.6 Honest observability

Do not claim that the runtime can observe information hidden by encryption or unavailable under the current Linux privilege and namespace model.
  • TCP metadata reveals addresses and ports, not encrypted application paths.
  • DNS telemetry may reveal queried names.
  • An explicit HTTP proxy can observe request metadata.
  • HTTPS CONNECT normally reveals destination host and port, not encrypted URL paths or bodies.
  • Process attribution for network traffic may require eBPF, cgroup hooks, netlink, or privileged access.
  • Inotify can overflow and does not provide perfect process attribution.
  • Separate stdout and stderr pipes do not establish perfect causal ordering across streams.

Core architectural rules

  • The daemon emits facts; the TypeScript SDK performs higher-level interpretation.
  • Protocol and telemetry types are versioned and shared, not manually duplicated.
  • Final event sequencing occurs at one deterministic point per execution.
  • Capture producers never synchronously depend on a slow remote sink.
  • Optional privileged collectors must not force the entire daemon to run permanently as root.
  • Large data is stored as artifacts and referenced by content-addressed identifiers.

8. Protocol, identifiers, and control API

8.1 Transport and framing

  • Primary transport: Unix domain socket.
  • Optional transport: stdio adapter for wrappers and deterministic integration tests.
  • Use explicit framing; never assume one read() equals one message.
  • Enforce a configurable maximum frame size before allocation.
  • When using stdio, protocol output goes only to stdout and human diagnostics go only to stderr.
  • Restrict socket permissions and validate Linux peer credentials where appropriate.
  • Handle stale socket paths without blindly unlinking arbitrary files.

8.2 Schema strategy

Evaluate protobuf versus a versioned framed JSON protocol. Record the choice in an ADR. The initial developer mode must remain inspectable. Rust and TypeScript must derive from or validate against a single schema source.

10. Process execution and lifecycle

10.1 Non-PTY execution

exec must support:

  • Executable and argument vector
  • Working directory
  • Validated environment overlay
  • Optional stdin stream
  • Separate stdout and stderr
  • Foreground and background execution
  • Timeout and cancellation
  • Execution/session association
  • Optional resource sampling
  • Graceful signal followed by forced termination
Do not accept an unsafe shell string as the only execution model. Shell execution must be explicit, for example /bin/bash -lc ....

10.2 Managed process record

Every managed command must track:

  • Logical ProcessId, OS PID, process group ID, and pidfd where available
  • Parent logical process where observable
  • Execution and session association
  • Command, arguments, working directory, and sanitized metadata
  • Start time, current state, cancellation handle, and completion notification
  • stdin/stdout/stderr stream offsets
  • Exit code, signal, reason, duration, and final timestamps

10.3 State machine

created → starting → running → terminating → exited
                     │             │
                     ├─────────────┴────→ signaled
                     └──────────────────→ failed

Reject invalid state transitions and unit-test the transition table.

10.4 Process groups, pidfds, subreaping, and PID 1

  • Create process groups deliberately so signals target the managed tree rather than only the direct child.
  • Investigate Linux pidfds to reduce PID-reuse races; provide a documented fallback.
  • Investigate PR_SET_CHILD_SUBREAPER to adopt and reap orphaned descendants.
  • If the daemon may run as PID 1, implement PID 1 signal handling and child reaping correctly.
  • Use close-on-exec on daemon-only descriptors.

10.5 Cancellation and shutdown sequence

  1. Stop accepting new work.
  2. Mark the process or execution as terminating.
  3. Send the configured graceful signal to the process group.
  4. Wait for the configured grace period.
  5. Send SIGKILL when necessary.
  6. Reap direct and adopted descendants.
  7. Drain/finalize streams according to policy.
  8. Emit final lifecycle and telemetry-loss events.

11. PTY and interactive session runtime

Implement Linux pseudoterminal-backed interactive sessions.

  • Allocate PTY master/slave pair.
  • Create a session and controlling terminal where appropriate.
  • Configure foreground process groups correctly.
  • Start the configured shell or command.
  • Capture PTY output and forwarded input as raw bytes.
  • Forward terminal resize events and dimensions.
  • Model terminal-generated signal behavior accurately.
  • Close and release all resources after normal or abnormal termination.
  • Record final process/session result.

SSH proxy integration boundary

  • The external proxy owns SSH authentication and edge transport.
  • sealantd owns the in-workspace PTY and shell/session lifecycle.
  • The proxy calls openSession, writeStdin, resizePty, and closeSession.
  • All proxied bytes must retain execution and session correlation.
  • Do not build a second internet-facing SSH authentication server unless an existing repository requirement demands it.
The daemon is not a terminal emulator. It captures and forwards PTY bytes.

13. Filesystem telemetry

Use a hybrid correctness strategy:

  1. Baseline snapshot
  2. Live event observation
  3. Final snapshot and diff
  4. Overflow/uncertainty recovery through rescan

Normalized events

file.modified
file.deleted
file.renamed
file.metadataChanged
file.watchOverflow
file.snapshotCompleted
file.diffAvailable

Snapshot metadata

  • Path and file type
  • Size and modification time
  • Permissions
  • Optional content hash
  • Symlink target

Required protections and behavior

  • Restrict observation to configured workspace roots.
  • Protect against symlink loops and path escape.
  • Handle special files and very large files safely.
  • Batch/coalesce editor temp-file patterns and repetitive metadata noise.
  • Apply ignore rules to generated trees such as node_modules.
  • Add watches for new directories and handle deleted directory trees.
  • Emit an explicit overflow event and trigger a rescan when the watcher cannot guarantee completeness.

Diffs and artifacts

  • Generate text patches only under configured type and size limits.
  • Store large patches and binary content as separate artifacts.
  • Use content-addressed artifact references.
  • Redact sensitive paths and content according to policy.
  • State whether rename detection is certain or inferred.
  • Do not claim reliable per-process attribution from inotify alone.

14.1 Capability-aware modes

14.2 Metadata mode

  • DNS observations where available
  • Local and remote addresses and ports
  • Protocol and direction
  • Open/close timestamps
  • Bytes sent/received where observable
  • Execution, session, and process attribution where observable

14.3 Proxy mode

  • Scheme where known
  • Host and port
  • HTTP method and URL/path only when observable
  • Response status and byte counts when observable
  • Timing and process/session attribution when possible
  • For HTTPS CONNECT, record destination host and port without pretending encrypted paths are known.
TLS interception, response-body capture, credential capture, or payload capture requires explicit product policy and a separate threat/privacy review. It is not enabled by default.

14.4 Privileged mode

  • Investigate eBPF, cgroup socket hooks, netlink/conntrack, packet capture, and transparent proxying.
  • Detect kernel features and Linux capabilities at startup.
  • Degrade cleanly when the backend cannot attach.
  • Do not make the entire daemon permanently privileged for an optional collector.
  • Consider a separate privileged collector process with a narrow IPC contract.

14.5 Source normalization

Emit an evidence event such as network.sourceObserved containing:

  • Hostname, resolved IPs, port, and scheme when known
  • URL/path only when actually observable
  • First and last observation times
  • Associated process/session identifiers
  • Observation method and confidence
  • Evidence event IDs

The daemon emits evidence. The TypeScript SDK decides whether that evidence represents a source used by an LLM task.

15. Telemetry pipeline

Implement a typed, bounded, observable pipeline with these stages:

  1. Event production
  2. Validation
  3. Redaction and policy
  4. Sequence assignment
  5. Batching
  6. Durable local spool
  7. Delivery
  8. Acknowledgement
  9. Retry or acknowledged deletion

Required properties

  • Event IDs and idempotent delivery keys
  • Bounded memory queues
  • Event priority classes
  • Explicit backpressure behavior
  • Batch size and time thresholds
  • Exponential backoff with jitter
  • Sink disconnect/reconnect handling
  • Duplicate-safe resend
  • Drop counters and degradation events
  • Graceful drain on shutdown

Priority guidance

PriorityExamplesExpected handling
Critical Control errors, runtime state changes, process start/exit, session close, drop events, corruption Spool before delivery; never silently discard.
Normal I/O chunks, filesystem mutations, network connections Bounded buffering, spill or configured degradation.
Low Repeated metrics, verbose watcher noise, optional payload details First candidates for coalescing or dropping with counters.

Assign final sequence values at one deterministic point per execution, not independently in every producer task.

18. Security and threat model

Assume workspace workloads can be buggy, hostile, evasive, or intentionally resource-exhausting.

Security design goals

  • Run the daemon under a protected identity and child workloads under a separate unprivileged UID where practical.
  • Use restrictive socket permissions and peer credential validation.
  • Exclude runtime secrets from child environments.
  • Minimize Linux capabilities and drop privilege after initialization.
  • Set close-on-exec on daemon-only descriptors.
  • Use validated paths and symlink-safe operations.
  • Do not construct shell commands through string concatenation.
  • Protect against fork bombs, output floods, event storms, and oversized protocol frames.
  • Consider PR_SET_DUMPABLE and ptrace restrictions where appropriate.

Required threat matrix

ThreatQuestions to answer
Child kills daemonDoes UID/capability separation prevent signaling? What is the degradation story?
Child reads secretsAre env vars, descriptors, files, proc entries, and logs isolated?
Unauthorized control clientHow are socket mode, ownership, and peer credentials validated?
Fake telemetry injectionCan workloads write directly to the spool or event transport?
Spool tamperingAre ownership, checksums, permissions, and corruption events sufficient?
Resource exhaustionWhat limits exist for output, forks, file events, sockets, memory, and disk?
Proxy bypassIs enforcement possible in the current network namespace and privilege mode?
Root workloadWhat guarantees become impossible when workload and daemon share a privileged namespace?
Do not describe same-namespace observation as tamper-proof unless the architecture actually provides a stronger privilege or isolation boundary.

Environment isolation

  • Create an explicit child base environment; do not blindly pass std::env::vars().
  • Remove telemetry credentials, control tokens, internal endpoints, spool keys, and debug secrets.
  • Validate environment keys.
  • Emit only allowlisted and redacted environment metadata.

Logging versus telemetry

  • Use structured tracing diagnostics for daemon operations.
  • Diagnostics go to stderr or a dedicated logging sink.
  • Product telemetry goes through the typed event pipeline.
  • Never duplicate secrets into diagnostic logs.

22. Implementation phases and exit gates

Phase 0 — Discovery and architecture

Deliver: repository analysis, requirements matrix, architecture, threat model, protocol ADR, validation plan, and workspace scaffold.

Exit gate:

  • Every requirement has an owner and test strategy.
  • Rust and TypeScript agree on protocol version 1.
  • No major subsystem lacks an explicit failure policy.
Phase 1 — Small end-to-end vertical slice

Implement: daemon startup, Unix socket, health, one non-PTY exec, stdout/stderr capture, exit event, graceful shutdown, and minimal TypeScript client.

Exit gate:

  • TypeScript starts the daemon, executes a command, streams events, receives the correct result, and shuts it down.
  • Invalid UTF-8 and NUL bytes round-trip correctly.
  • Protocol output is never mixed with diagnostics.
Phase 2 — Correct process lifecycle

Implement: registry, logical IDs, process groups, signals, timeout, grandchild cleanup, subreaper/PID 1 behavior, stdin, background commands, and optional resource metrics.

Exit gate:

  • No zombies or managed-process orphans in the process matrix.
  • SIGTERM escalation works.
  • PID-reuse-sensitive operations use pidfds or a documented safe fallback.
  • Runtime shutdown works with active process trees.
Phase 3 — PTY and session runtime

Implement: PTY shell, input/output capture, resize, signal behavior, session close, and SSH-proxy API.

Exit gate:

  • Interactive shell and full-screen application work.
  • Resize propagates correctly.
  • Input/output replay is binary-safe.
  • Resources are released after normal and abnormal exits.
Phase 4 — Durable telemetry pipeline

Implement: bounded queues, priorities, sequencing, spool, checksums, recovery, retry, acknowledgements, and degradation/drop events.

Exit gate:

  • Slow sinks do not create unbounded memory growth.
  • Restart replays unacknowledged events safely.
  • Corrupt and partial records are deterministic.
  • Critical event loss is never silent.
Phase 5 — Filesystem telemetry

Implement: baseline snapshot, live watcher, coalescing, final diff, artifacts, and overflow recovery.

Exit gate:

  • Create/modify/delete/rename cases pass.
  • Editor temp-file patterns are normalized reasonably.
  • Overflow emits an event and triggers rescan.
  • Symlink and path-boundary tests pass.
Phase 6 — Network telemetry

Implement: capability detection, metadata mode, proxy mode, supported privileged backend, source normalization, and degraded fallback.

Exit gate:

  • Local DNS, HTTP, HTTPS CONNECT, TCP, and UDP fixtures produce correct evidence.
  • Attribution uncertainty is represented honestly.
  • Missing privilege never crashes the daemon.
Phase 7 — Security and hardening

Implement: identity strategy, environment isolation, peer validation, capability minimization, redaction, limits, and feature kill switches.

Exit gate:

  • Threat-model tests pass.
  • Daemon secrets do not leak through env or inherited descriptors.
  • Unauthorized clients are rejected.
  • Protocol and spool fuzzers find no obvious crashes.
Phase 8 — Packaging and final integration

Deliver: multi-architecture artifacts, BuildKit stage, TypeScript package, examples, operational docs, benchmark report, and complete requirements traceability.

23. Required validation loop

  1. Define one concrete behavior and its acceptance test.
  2. Update the typed protocol and requirement matrix.
  3. Write focused tests before or alongside implementation.
  4. Implement the smallest complete vertical slice.
  5. Run formatting, linting, unit, and type checks.
  6. Run subsystem integration tests.
  7. Run container-level tests.
  8. Inject realistic failures.
  9. Request an independent high-risk review where appropriate.
  10. Fix all blocking findings.
  11. Update architecture, protocol, limitations, and requirement status.
  12. Commit only after the acceptance gate passes.
A requirement is not complete because code exists. It is complete when observable behavior matches the contract under normal, failure, and shutdown scenarios.

24. Mandatory adversarial test matrix

Process tests
  • Command not found
  • Immediate exit, exit 0, and exit 42
  • Killed by SIGTERM and SIGKILL
  • Child ignores SIGTERM
  • Child spawns grandchildren
  • Double-fork and adopted descendant
  • Grandchild inherits stdout/stderr descriptors
  • Large stdout and large stderr
  • Concurrent stdout and stderr
  • NUL bytes and invalid UTF-8
  • stdin close and broken pipe
  • Timeout and runtime shutdown during execution
  • PID reuse risk and zombie detection
PTY tests
  • Interactive shell
  • Echoed and raw input
  • ANSI and binary-safe output
  • Full-screen terminal application
  • Resize
  • Ctrl+C and terminal signal behavior
  • Session close and unexpected child exit
  • Proxy disconnect
  • Recording and replay
Telemetry and spool tests
  • Slow and disconnected sink
  • Queue full and spool full
  • Disk full and permission failure
  • Corrupt and truncated records
  • Duplicate delivery and restart replay
  • Sequence ordering and gaps
  • Drop reporting and critical-event preservation
Filesystem tests
  • Create, modify, delete, and rename
  • Directory create/delete
  • Symlink and symlink loop
  • Large and binary files
  • Ignore rules
  • Editor temp-file save pattern
  • Watcher overflow
  • Path traversal and workspace escape attempts
Network tests
  • DNS lookup
  • HTTP request
  • HTTPS CONNECT
  • TCP and UDP traffic
  • Connection failure
  • Proxy bypass attempt
  • Missing Linux capabilities
  • Privileged collector startup failure
  • Source normalization and unknown hostname
  • Multiple IPs for one host
Security tests
  • Child tries to read daemon environment
  • Child tries to access the control socket
  • Child sends signals to the daemon
  • Child attempts ptrace where applicable
  • Oversized and malformed control frames
  • Invalid path and signal input
  • Secret-like output redaction
  • Runtime-only environment leakage
  • Excessive forks, output, and file events

25. Performance, resource, and soak validation

Create repeatable measurements rather than making speculative performance claims.

  • Stream a large amount of stdout to a fast sink.
  • Repeat with a deliberately slow sink.
  • Verify resident memory remains bounded.
  • Run many concurrent short-lived processes.
  • Run multiple long-lived PTY sessions.
  • Generate filesystem-event bursts and watcher overflow.
  • Generate many network connections.
  • Exercise spool rotation and replay over an extended run.
  • Repeatedly shut down under active load.
  • Record CPU, memory, disk, queue, and latency measurements with the exact configuration.

27. Reporting behavior

At the end of every phase, report:

  1. What was built
  2. Which requirements were completed
  3. Which tests passed and failed
  4. Architecture decisions made
  5. Security and observability limitations
  6. The next highest-risk item

Clearly distinguish:

  • Implemented and validated
  • Implemented but environment-dependent
  • Degraded fallback implemented
  • Designed but not implemented
  • Blocked by kernel, runtime, or privilege capabilities
